Algebraic Structure

In mathematics, an algebraic structure is defined by three things:

  • a set
  • one or more operations on the elements of the set
  • a list of laws that the operations must follow

For example, all rational numbers (the set), together with the operation + (addition) and ⋅ (multiplication), and the laws:

  • x + y = y + x
  • x + 0 = x
  • x(y + z) = xy + xz
  • … seven other laws that nobody cares about

forms an algebraic structure called field.

You learned how to do addition and multiplication on rational numbers before knowing that they form a field. The point is: the term “algebraic structure” itself is not interesting, (at least not to us programmers), it’s just a terminology. What’s interesting is how the underlying operations work and how they relates to each other by the laws.

The reason these structures exist is that, many different things have the same underlying structure, it’s better to make an abstraction and to study them once and for all, rather than studying them separately and reinventing the same wheel again and again.

There is no doubt that these abstractions will introduce a barrier to beginners: you must learn the concepts before you can use them. But this is definitely worth it, once you learned the concepts and used it a few times, the next time you see something similar, you will have a rough guess about how things should work - without looking into the documentations. Don’t worry if all of these sound blurry, after the project optparse-applicative, you will see what I mean.

Algebraic structures in Haskell are represented via tyepclasses, such as semigroups, monoids and mondas. However, unlike types, the laws that the algebraic structure should obey is not enforced by GHC, it’s the implementor’s job to make sure that his implementation satisfies those laws.

Tip: if you have trouble understanding a definition, don’t stuck on it for too much time, do some examples and write some concrete codes, then come back to it, you will have a easier time.
